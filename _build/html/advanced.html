Basic concepts¶

django_roles implements it’s core roles with django.contrib.auth.models.Group. This means that django.contrib.auth must be listed in INSTALLED_APPS.

This could be an extra effort if you just want to grant a django.contrib.auth.models.User access to a view. For example in case where only a particular user must be the only one in access a particular view. To solve this you can create a new django.contrib.auth.models.Group and named it in such a way you reflect this particular case.

In general there are, from security point of view, two kind of applications:

  • Simple: This kind of sites requires user to be logged in some views. This is the case when @login_required is used.
  • Heavy: This kind of sites requires user to be logged in almost all views. The use of @login_required became impractical or at least tedious.

Django Roles could work in to ways depending in site’s security requirements as explained: simple or heave:

  • With function/decorator: For this cases Django Roles provides a decorator or function to check security.
  • With middleware: When a site requires user to be logged in almost all views. Django Roles offers a middleware to free you of the need of using the mentioned decorator.

For any of the cases (function/decorator or middleware) Django Roles is used equally.

Namespaces and View Name¶

When creating a roles.models.ViewAccess object, the value of view attribute can be:

  • view_name
  • app_name:view_name
  • namespace:view_name
  • nest_namespace:namespace:view_name

As this are the only possible value for view attribute, a requirement of django_roles is to use view names and app_name in urls.py files. Read more about Naming URL in official Django project documentation Naming URL.
